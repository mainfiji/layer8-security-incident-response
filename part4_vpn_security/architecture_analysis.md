# Анализ архитектуры VPN-шлюза Layer 8

## Общие сведения

В компании используется VPN-шлюз на базе WireGuard, развёрнутый на хосте `srv_wg_portal` (IP 192.168.4.20) в сегменте DMZ. Шлюз интегрирован с LDAP для аутентификации пользователей.

### Текущая конфигурация:
- **Протокол:** WireGuard
- **Порт:** UDP 51820 (стандартный)
- **Интерфейс:** wg0 (подсеть 10.10.10.0/24)
- **Клиенты:** все получают IP из одного пула, один и тот же набор маршрутов
- **Маршрутизация:** AllowedIPs = 0.0.0.0/0 (весь трафик через VPN)

---

## Детальный анализ безопасности

### 1. Может ли пользователь изменить конфигурацию и получить доступ к дополнительным сегментам?

**Ответ: Да.**

Пользователь может изменить параметр `AllowedIPs` в своём файле конфигурации WireGuard (обычно `/etc/wireguard/wg0.conf`). Если он добавит дополнительные подсети (например, `192.168.1.0/24`, `192.168.5.0/24`), трафик к ним пойдёт через туннель.

**Условия:**
- На шлюзе должны быть маршруты до этих подсетей (они есть, так как шлюз транзитный).
- На межсетевом экране `fw` нет ограничений по источникам для этих подсетей.

**Риск:** Пользователь может получить доступ к сегментам, к которым не имеет права (например, к серверному сегменту 192.168.1.0/24).

---

### 2. Возможно ли разделение прав доступа между пользователями?

**Ответ: Нет.**

Все клиенты VPN находятся в одной подсети `wg0` (10.10.10.0/24). На межсетевом экране `fw` нет правил, различающих клиентов по IP-адресам (все пакеты приходят с IP-адреса шлюза или из одной подсети).

**Отсутствуют:**
- Группы пользователей в LDAP, применяемые к маршрутизации
- Политики на основе учётных записей
- Разные VPN-интерфейсы для разных ролей

**Риск:** Невозможность ограничить доступ, например, разработчики имеют доступ к серверному сегменту так же, как и администраторы.

---

### 3. Изолированы ли пользователи друг от друга?

**Ответ: Нет.**

Все клиенты находятся в одной подсети и могут взаимодействовать напрямую (ping, открытые порты). Никаких правил на шлюзе для блокировки трафика между клиентами нет.

**Риск:** Атакующий, скомпрометировавший один хост, может легко перейти на другой хост внутри VPN-сети (горизонтальное перемещение).

---

### 4. Архитектурные недостатки (резюме)

| Недостаток | Последствие |
|------------|-------------|
| Все клиенты в одной подсети | Нет изоляции, горизонтальное перемещение |
| Общий набор маршрутов | Нет ролевого доступа |
| Отсутствие политик на основе источников | Легко обходится изменением `AllowedIPs` |
| Нет централизованного аудита действий VPN | Сложность расследования инцидентов |
| Использование только IP-фильтрации (нет аутентификации на уровне приложений) | Недостаточный контроль |

---

### 5. Возможность интеграции с Wazuh

Логи WireGuard (на самом шлюзе) можно передавать в Wazuh для мониторинга подключений, активности и ошибок. Однако в текущей архитектуре это не реализовано.

---

## Заключение

Текущая архитектура VPN не соответствует принципу минимальных привилегий и нулевого доверия. Требуется кардинальное переосмысление с внедрением ролевой модели и изоляции клиентов.
