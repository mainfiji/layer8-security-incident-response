# Часть 4. Анализ архитектуры VPN

## Обзор

В компании используется VPN-шлюз на базе WireGuard, развёрнутый на хосте `srv_wg_portal` в сегменте DMZ. Проведён анализ текущего архитектурного решения на предмет безопасности.

## Ключевые выводы

### 1. Может ли пользователь изменять конфигурацию?

**Ответ: Да.**

В WireGuard пользователь может изменить параметр `AllowedIPs` в своём файле конфигурации `/etc/wireguard/wg0.conf`, добавив любые дополнительные подсети. Трафик к ним пойдёт через туннель, если маршрутизация на шлюзе разрешена.

**Риск:** Пользователь может получить доступ к сегментам, к которым не имеет права.

---

### 2. Возможно ли разделение прав доступа между пользователями?

**Ответ: Нет.**

Все клиенты VPN находятся в одной подсети `wg0` (10.10.10.0/24). Межсетевой экран `fw` не различает клиентов по их IP-адресам.

**Риск:** Невозможность ограничить доступ по ролям (например, разработчики имеют доступ к серверному сегменту так же, как и администраторы).

---

### 3. Изолированы ли пользователи друг от друга?

**Ответ: Нет.**

Все клиенты находятся в одной подсети и могут взаимодействовать напрямую (ping, открытые порты).

**Риск:** Горизонтальное перемещение атакующего между пользователями после компрометации одного хоста.

---

## Выявленные архитектурные недостатки

| Недостаток | Последствие |
|------------|-------------|
| Все клиенты в одной подсети | Нет изоляции, горизонтальное перемещение |
| Общий набор маршрутов | Нет ролевого доступа |
| Отсутствие политик на основе источников | Легко обходится изменением `AllowedIPs` |
| Нет централизованного аудита действий VPN | Сложность расследования инцидентов |
| Использование только IP-фильтрации | Недостаточный контроль доступа |

## Рекомендации по улучшению

1. **Внедрение SSL VPN с поддержкой ролевых политик**
   - OpenVPN Access Server, WireGuard + FreeRADIUS, StrongSwan

2. **Разделение VPN-пулов по ролям**
   - `wg_dev` — разработчики (доступ только к сегменту разработки)
   - `wg_admin` — администраторы (доступ ко всем сегментам)
   - `wg_users` — обычные сотрудники (доступ только к файловому серверу)

Выводы
Текущая архитектура VPN не соответствует современным требованиям безопасности. Отсутствие разграничения доступа и изоляции клиентов создаёт риск горизонтального перемещения и несанкционированного доступа к ресурсам. Рекомендовано внедрение решений с поддержкой ролевой модели и Zero Trust принципов.
