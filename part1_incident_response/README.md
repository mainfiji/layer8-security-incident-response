## Инцидент №1. Массовые блокировки пользователей

### Описание проблемы

Клиенты жаловались на ошибки `503 Service Unavailable` и `429 Too Many Requests` при работе с сайтом. При анализе журналов было обнаружено:

**В access.log nginx:**
192.168.2.15 - - [12/Dec/2025:10:23:45 +0000] "GET /candidates?page=2 HTTP/1.1" 200 15432
192.168.2.15 - - [12/Dec/2025:10:23:46 +0000] "GET /candidates?page=3 HTTP/1.1" 200 15800
... (30+ запросов за 2 секунды)
192.168.2.15 - - [12/Dec/2025:10:23:48 +0000] "GET /candidates?page=15 HTTP/1.1" 503 0

text

**В журнале ModSecurity:**
ModSecurity: Access denied with code 503 (phase 2). Operator GE matched 60 at IP: rate.
[msg "Too Many Requests"] [data "current request rate 62/min - exceeded limit 60/min"]

text

### Причина

Ложное срабатывание механизма ограничения частоты запросов. Легитимные пользователи (сотрудники кадрового агентства) делали большое количество запросов (активный поиск, просмотр анкет), что превышало установленный лимит 60 запросов в минуту.

### Решение

Настроена гибкая политика rate limiting с исключениями для доверенных сегментов:

```nginx
limit_req_zone $binary_remote_addr zone=dynamic:10m rate=120r/m;
limit_req_zone $binary_remote_addr zone=static:10m rate=300r/m;

geo $limited_ip {
    default 1;
    192.168.2.0/24 0;   # Пользовательский сегмент — не ограничивать
    192.168.6.0/24 0;   # Сегмент разработки — не ограничивать
}

server {
    location /candidates {
        if ($limited_ip = 0) {
            set $rate_limit_bypass 1;
        }
        limit_req zone=dynamic burst=30 nodelay;
    }
}

Результат
Легитимные пользователи перестали блокироваться, при этом защита от DDoS сохранилась для внешних IP-адресов.

Инцидент №2. Подозрительная активность на хосте pc_vkrutikov
Обнаружение
В Wazuh поступил алерт о множественных неудачных попытках SSH-входа (брутфорс) с внешнего IP 185.130.5.44 на хосте pc_vkrutikov.

Действия
Анализ: проверены логи /var/log/auth.log — обнаружены десятки неудачных попыток за последние 5 минут.

Принятие решения: активность явно вредоносная, хост необходимо изолировать.

Изоляция: через панель управления WireGuard (srv_wg_portal) выполнен отключение пира, изолирующее хост от корпоративной сети.

Уведомление: администратору инфраструктуры и CISO передана информация для дальнейшего расследования.

Взаимодействие с командами
Администратор инфраструктуры — проверка состояния хоста, снятие дампа памяти и диска.

CISO — принятие решения о масштабе реагирования.

Служба техподдержки — информирование сотрудника о временной недоступности ресурсов.

Инцидент №3. Подозрительная активность на хостах srv_waf и srv_ldap
Обнаружение
Обнаружены множественные алерты о подозрительных действиях:

Модификация системных файлов (/etc/passwd, /etc/shadow)

Создание новых пользователей

Необычные сетевые соединения на высокие порты

Действия
Немедленная изоляция: серверы полностью изолированы от сети на межсетевом экране, кроме управляющих соединений (для сохранения возможности расследования).

Сбор доказательств: анализ логов Wazuh, auth.log, логов LDAP через управляющую сеть.

Выявлен путь атаки: злоумышленник получил доступ через уязвимость в веб-приложении на srv_waf, повысил привилегии, установил бэкдор, затем перешёл на srv_ldap через скомпрометированные учётные данные.

Подготовка к восстановлению: серверы выведены из эксплуатации, заменены чистыми экземплярами из резервной копии.

Взаимодействие с командами
Администратор веб-сервисов — анализ конфигурации сайта, поиск уязвимости.

Администратор инфраструктуры — восстановление серверов из бэкапов.

Аналитик SOC — углублённый анализ артефактов, выявление IOC.

Выводы
Отработаны навыки реагирования на инциденты разных типов:

Ложные срабатывания СЗИ (требуется тонкая настройка)

Брутфорс-атаки (требуется изоляция и последующее расследование)

Компрометация критичных серверов (требуется полная изоляция, сбор доказательств, восстановление из бэкапов)
