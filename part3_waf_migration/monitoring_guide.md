# Инструкция: Мониторинг событий безопасности WAF BunkerWeb

## 1. Введение

Данная инструкция предназначена для администраторов и аналитиков SOC, отвечающих за мониторинг и реагирование на инциденты безопасности, связанные с веб-приложениями, защищёнными с помощью WAF BunkerWeb.

## 2. Доступ к веб-интерфейсу

BunkerWeb предоставляет веб-интерфейс (UI) для мониторинга и управления. По умолчанию доступен по порту **9443** (HTTPS).
https://<IP_адрес_WAF>:9443

text

Учётные данные задаются при установке (переменные `WEB_UI_LOGIN` и `WEB_UI_PASSWORD`).

## 3. Основные разделы мониторинга

### 3.1 Dashboard (Панель управления)
- Общее количество запросов (всего, успешных, заблокированных)
- График нагрузки (запросы в секунду)
- Топ-5 атакующих IP
- Типы атак (круговая диаграмма)
- Статус бэкендов

### 3.2 Attacks (Атаки)
Детализированная информация о каждом заблокированном запросе:
- Время, IP-адрес атакующего, тип атаки, URI
- Фильтрация по дате, IP, типу атаки, правилу
- Просмотр полных заголовков и тела запроса
- Экспорт в CSV или JSON

### 3.3 Requests (Запросы)
Все запросы, прошедшие через WAF (разрешённые и заблокированные):
- Поиск конкретных запросов
- Анализ поведения ботов
- Проверка ложных срабатываний

### 3.4 Logs (Журналы)
Просмотр логов WAF в реальном времени (хвост логов).
Настройка уровня детализации (info, warn, error).

## Обзор

В компании используется VPN-шлюз на базе WireGuard, развёрнутый на хосте `srv_wg_portal` в сегменте DMZ. Проведён анализ текущего архитектурного решения на предмет безопасности.

## Текущая архитектура
Интернет → srv_wg_portal (DMZ, 192.168.4.0/24) → wg0 (VPN-подсеть)
↓
Все клиенты в одной подсети
↓
Доступ ко всем сегментам

text

**Особенности:**
- Все VPN-клиенты получают IP из одной подсети (`wg0`)
- Все клиенты имеют одинаковый набор маршрутов
- Используется протокол WireGuard (UDP 51820)
- Шлюз интегрирован с LDAP для аутентификации

---

## Анализ безопасности

### Вопрос 1. Может ли пользователь изменить конфигурацию?

**Ответ: Да.**

В WireGuard на стороне клиента в секции `[Interface]` задан IP-адрес из пула VPN, а в секции `[Peer]` указан `AllowedIPs = 0.0.0.0/0`.

Пользователь может изменить `AllowedIPs`, добавив любые другие подсети (например, `192.168.1.0/24`, `192.168.5.0/24`), и трафик к ним также пойдёт через VPN-туннель, если маршрутизация на шлюзе разрешена.

**Риск:** Пользователь может получить доступ к сетям, к которым не имеет права.

### Вопрос 2. Возможно ли разделение прав доступа?

**Ответ: Нет.**

Все VPN-клиенты попадают в одну сеть `wg0` и получают одинаковый набор маршрутов. Межсетевой экран `fw` не различает пользователей VPN по их IP-адресам (все они приходят с одного IP шлюза) и не применяет политики на основе учётных данных.

**Риск:** Невозможность ограничения доступа по ролям (например, разработчики → только серверный сегмент, бухгалтерия → только учётные системы).

### Вопрос 3. Изолированы ли пользователи друг от друга?

**Ответ: Нет.**

Все клиенты находятся в одной широковещательной домене. Они могут взаимодействовать друг с другом напрямую (ping, доступ к открытым портам), если это не запрещено правилами на самих клиентах или на промежуточных устройствах.

**Риск:** Горизонтальное перемещение атакующего между пользователями после компрометации одного хоста.

## Выявленные архитектурные недостатки

| Недостаток | Описание | Риск |
|------------|----------|------|
| Отсутствие сегментации пользователей | Все имеют одинаковый уровень доступа | Повышенная поверхность атаки |
| Невозможность применения политик на основе ролей | Нельзя ограничить доступ по группам пользователей | Нарушение принципа минимальных привилегий |
| Отсутствие изоляции между клиентами | Клиенты могут взаимодействовать друг с другом | Горизонтальное перемещение атакующего |
| Управление доступом на уровне IP | Легко обходится подменой конфигурации | Недостаточный контроль доступа |
| Нет централизованного аудита действий внутри VPN | Отсутствие детальных логов активности пользователей | Сложность расследования инцидентов |

## Рекомендации по улучшению

### 1. Внедрение SSL VPN с политиками
Заменить WireGuard на решение с поддержкой аутентификации по LDAP и назначением прав после подключения (например, OpenVPN Access Server, WireGuard с FreeRADIUS).

### 2. Разделение VPN-пулов по ролям
Создать несколько виртуальных интерфейсов:
- `wg_dev` — для разработчиков (доступ только к серверному сегменту)
- `wg_admin` — для администраторов (доступ ко всем сегментам)
- `wg_users` — для обычных сотрудников (доступ только к файловому серверу)

На межсетевом экране разрешить доступ из соответствующих подсетей только к нужным сегментам.

### 3. Изоляция клиентов на уровне nftables
Добавить правила на шлюзе, запрещающие прямой трафик между IP-адресами клиентов:
```bash
nft add rule inet filter forward ip saddr 10.10.10.0/24 ip daddr 10.10.10.0/24 drop
4. Внедрение Zero Trust Network Access (ZTNA)
Использовать прокси-агенты на конечных точках с проверкой прав доступа на уровне приложений.

5. Система управления доступом на основе атрибутов (ABAC)
Интеграция с RADIUS и динамическими списками доступа на маршрутизаторах/коммутаторах.

6. Централизованный аудит
